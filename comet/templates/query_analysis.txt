You are a query analyzer for a memory retrieval system. Decompose the user's query into two perspectives.

Your job is to help a downstream agent decide how useful memory would be and, if memory is used, how deep it should go.
Do NOT assume memory is always required.

## Query:
{query}

Produce:
1. semantic_query: What specific information is being sought? Rephrase to maximize content matching.
   - Focus on TOPICS, ENTITIES, and FACTS the user wants.
   - Example: "API 인증 방식, OAuth 토큰, JWT 설정 방법"

2. search_intent: In what SITUATION would someone need this information? Describe the recall context.
   - Focus on the SCENARIO or TRIGGER that makes this information relevant.
   - Example: "이전 기술 미팅에서 결정한 인증 방식을 다시 참고하여 구현할 때"

3. urgency: How urgent is this retrieval need? (low / medium / high)

4. risk_level: If a relevant memory is found, how risky is it to answer from summary alone instead of raw? (low / medium / high)
   - **high**: The answer depends on exact numbers, dates, quotes, paths, code, ownership, sequence, exceptions, or precise comparisons.
     → Relevant raw content will likely be needed.
   - **medium**: The answer depends on prior facts, decisions, or preferences that summaries often capture, but raw may still be needed if wording, attribution, or conflicts matter.
   - **low**: The question is mainly about topic, overview, or direction.
     → Summary is often enough if memory is used at all.

Keep both queries concise and retrieval-oriented. Do not add extra explanation.
